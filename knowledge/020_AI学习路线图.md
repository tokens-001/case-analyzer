> 每个概念：一句话 → 你项目里对标什么 → 要不要现在学

## 第二层：原理知道但没亲手实现（4项，暂不学）

### RAG — 检索增强生成

**一句话**：先查外部资料，把结果塞进提示词，让 AI 参考着回答。不是靠 AI 的记忆。

**你项目的对标**：`search_law_database()` 在法条库里查到 AI 引用的法条原文 → 展示给人对照。你走的是"事后校验"路线（AI 生成后对答案），传统 RAG 走"事前增强"路线（AI 生成前喂资料）。方向不同，目的相同。

**什么时候学**：法条库大到文件名匹配漏东西，或者想自动把法条原文注回 AI 提示词时。

---

### Agent — 智能体

**一句话**：能自己决定"下一步做什么"的 AI。不是被动回答问题，是主动规划步骤。

**你项目的对标**：`/analyze` 四轮流程是预设好的。真正的 Agent：你告诉它"分析判例"，它自己决定先调哪个 skill、后调哪个，每一步是 AI 选的不是代码定的。

**为什么现在不需要**：法律场景需要确定性。验证层每一步都不能跳过，Agent 可能省成本省略校验。

**什么时候学**：有 50+ 个技能、用户需求各异，没法预设所有流程时。

---

### Tool Calling — 工具调用

**一句话**：AI 能调用外部函数（查数据库、发请求），不只是生成文字。

**你项目的对标**：你的代码决定何时调 `search_law_database()`。Tool Calling 反过来：AI 说"我需要查法条库了" → 自动调函数 → 拿到结果继续分析。

**为什么现在不需要**：校验层顺序是固定的，不需要 AI 自己决定。

**什么时候学**：技能多到没法预设流程，需要 AI 自己判断该调哪个工具时。

---

### MCP — 模型上下文协议

**一句话**：一套标准接口，不管什么模型（GPT/Claude/DeepSeek），用同一种方式接入外部工具。

**你项目的对标**：只接了 DeepSeek 一个模型，工具都是直接 import Python 函数。MCP 的场景是 GPT + Claude + DeepSeek 三个模型都想调 `skills/legal/`，不用写三套接口。

**什么时候学**：有 2+ 个模型且共享同一套技能库时。

## 第三层：需要学的（5项，按优先级排列）

### P1 — Embedding / 向量嵌入

**一句话**：把一段文字变成一串数字（向量）。相似内容的向量也相似。

**对比你现在**：
```
现在（关键词）: if "民法典" in 文件名 → 找到
以后（向量）:   用户搜"劳动纠纷" → 找到写"解除劳动合同"的判例
```

**什么时候该学**：法条库超 50 部法律，或判例超 100 份，文件名匹配开始漏东西。当前量不需要。学的时候先了解 `text-embedding-3-small` 这个 API。

---

### P2 — Evaluation（LLM 评估）

**一句话**：不只看 AI 输出有没有事实错误（验证层），还看输出有没有用、好不好。

**你已经在做但不知道叫这个名字的事**：
- 法条校验 → 检查引用是否真实
- 溯源 → 检查段落号是否编造
- 可信度评分 → 0-100 分 + 加减分明细
- 风险列表 → 红色 danger / 黄色 warning
- 验证分析结果 → 字数检查 + 法条引用检查

这些全都属于 LLM Evaluation。你做的不是"让 AI 自己评分"，而是"用独立规则评分"——这恰恰是最可靠的 Evaluation 方式。

**为什么这是你的优势**：大多数 AI 产品停在"生成"。你比别人多了一层"验证"。把验证层系统化，就是你的 Evaluation 框架。

**什么时候深入**：第二个项目需要同样的验证层设计时。抽象出"验证层设计 Checklist"（外部知识引用→需要引用真实性校验；AI 生成步骤→需要溯源；评分→需要可解释明细）。

---

### P3 — OpenAI API

**一句话**：用代码直接调 GPT，不是 ChatGPT 网页。按量付费，和订阅分开。

**为什么现在需要**：你讨论的核心问题是多 Agent 协作。多 Agent 协作的前提是有多个独立模型可以交叉验证。你已有 DeepSeek API，加上 OpenAI API 就是两个独立视角。再加上 Claude（通过 Claude Code）就是三个。

**最低成本**：注册 OpenAI 账号 → 充 $5 → 拿到 API key → `pip install openai` → 3 行代码调 GPT-4o。

---

### P4 — Semantic Search / 语义搜索

**一句话**：Embedding + 检索 + 排序。搜"解雇"能找到"解除劳动合同"。

**对比你现在**：`过滤历史()` 用 `.includes(kw)` 匹配。

**什么时候该学**：需要给判例助手加语义搜索历史判例时。

---

### P5 — LangChain / LangGraph（降低优先级）

**一句话**：LLM 编排框架。把多个 AI 调用串成流程。

**⚠️ GPT 修正**：你现在是先理解编排再接触框架，不是靠框架理解编排。LangChain 对你来说是便利工具，不是核心能力。等手写 `ThreadPoolExecutor` 真的管理不过来再学。

### P6 — CrewAI / AutoGen（降低优先级）

**一句话**：多 Agent 编排框架。

**⚠️ GPT 修正**：agent_bus 已经解决了你的核心问题（共享记忆 + 任务流转）。CrewAI 解决的问题是"Agent 太多手写不动了"。3 个角色不需要。

## 第四层：完全没概念的（4项，碰到了再学）

| 概念 | 一句话 | 触发条件 |
|------|--------|---------|
| Streaming / SSE | AI 一个字一个字往外蹦，不用等 35 秒 | 用户嫌等太久 |
| Token 预算管理 | 控制输入+输出不超过模型上下文窗口 | 判例文字超 4 万字（DeepSeek 64K token） |
| Evaluation | 量化评估"AI 分析好不好"（不是事实性校验，是有用性评估） | 想知道 AI 分析质量是不是在改善 |
| Guardrails | 限制 AI 不乱说（敏感信息过滤、伦理边界） | 判例包含个人隐私信息 |

## 学习优先级总结（经 GPT 修正）

```
现在不需要动的：  9项 ✅  + 4项 🟡  = 13项（已经在项目里用着或知道原理）

下一个该学的：
  P1 Embedding  → 法条库 50+ 部时

  P3 OpenAI API → 双模型交叉验证的前提
  P4 Semantic Search → 和 P1 一套

低优先级（便利工具，非核心能力）：
  P5 LangChain → 先手写编排再学框架
  P6 CrewAI   → agent_bus 够用，3 角色不需要框架

碰到了再学：      4项 ⬜（Streaming / Token / Guardrails 从第四层移到这）
```

## 判断规则

以后听到任何新概念，顺序问自己三句：

1. **它管什么层？**（Skill / Workflow / Knowledge / Model / 都不属于）
2. **跟我项目里哪个东西对标？**（答不上来 = 真盲区）
3. **我现在的项目碰到它要解决的问题了吗？**（没碰到 = 不现在学）
