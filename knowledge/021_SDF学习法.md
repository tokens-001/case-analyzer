# 021 — SDF 学习法

> 不是给程序员的。是给 AI 协作者/产品架构师的。
> 核心原则：永远从大到小。不要从代码开始。

---

## 三层分析法

### S — System（系统层）— 先画全局

不看代码。先回答：**这个东西在整个系统里干什么？**

```
用户 → 输入判例 → AI分析 → 验证 → 评分 → 展示
```

系统层没懂，不看代码。因为一定看不懂。

### D — Data Flow（数据流）— 再看进出

每个模块只问两句：**输入是什么？输出是什么？**

```
verify_laws:
  输入：《民法典》第577条
  处理：法条库比对
  输出：存在/不存在 + 有效/失效

trace_citations:
  输入：AI引用段落号
  处理：段落号校验
  输出：风险列表
```

### F — Function（函数）— 最后看代码

看函数时只问四句：
1. 为什么存在？
2. 输入是什么？
3. 输出是什么？
4. 被谁调用？

## 三阶段学习路线

### 第一阶段（现在）：能讲清判例助手怎么运行

每天一个模块。不学新技术。每个模块回答五问：
1. 属于哪层（Skill / Workflow / Validation / Product）
2. 输入是什么
3. 输出是什么
4. 依赖谁
5. 谁依赖它

### 第二阶段（反馈期）：能看懂 Claude 改了什么

每次改动画两张图：
```
修改前 → 修改后 → 变化在哪
```

### 第三阶段（OpenAI API 后）：理解 AI 系统组成

只学四个核心概念：API / Embedding / RAG / Tool Calling。别超过四个。

---

## 核心原则

> 不追求会写新的东西，追求能解释已有的东西。

判例助手已经足够复杂。把 前端→路由→技能→验证层→反馈层→存储层 完整讲清楚，以后 Claude 写 1000 行、5000 行、1 万行代码，你都不会失控。

---

## 知识资产生产线

```
项目 → 发现 → learning → 重复出现 → pattern → 跨项目验证 → Product OS
```

不是无意识的。每条 learning 都知道自己在哪个阶段。
