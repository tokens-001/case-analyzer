## 一、概念对照表

### Skill（技能）

```
通俗理解：一个独立的能力模块，输入→处理→输出
你的项目：skills/legal/*.py（13个文件，每个一个能力）
为什么需要：同一个"法条校验"能力，网页用、CLI用、Agent用，不需要重写
不用会怎样：所有逻辑堆在 app.py 里，换个场景就得复制粘贴
```

### Workflow（工作流）

```
通俗理解：多个技能的执行顺序和依赖关系
你的项目：/analyze 路由里的四轮调度（4并行→1串行→5并行→本地校验）
为什么需要：技能是原子，工作流是分子。不同场景需要不同的技能组合
不用会怎样：每次手动决定先调哪个再调哪个，容易漏步骤
```

### Orchestration（编排）

```
通俗理解：管理和调度多个技能/工作流执行的控制器
你的项目：ThreadPoolExecutor + /analyze 路由的调度逻辑
为什么需要：处理依赖关系（总结依赖前四个分析结果）、并行执行、结果组装
不用会怎样：串行等所有结果，35秒变2分钟；依赖关系靠人记
```

### Validation Layer（验证层）

```
通俗理解：AI 生成之后，用非 AI 方式校验生成内容是否可信
你的项目：verify_laws.py + trace_citations.py + score_analysis.py
为什么需要：AI 会编法条、编段落号、编引用。独立校验是唯一可信路径
不用会怎样：用户看到 AI 编的假法条，整个产品信任崩塌
```

### RAG（检索增强生成）

```
通俗理解：AI 生成前，先从外部知识库找相关证据，塞进提示词里参考
你的项目：data/laws/ 法条库 → search_law_database() → 与AI引用逐条对照
为什么需要：模型不知道哪条法废止了、哪条是现行有效的。外部知识库提供证据
不用会怎样：AI 引用一个三年前已废止的法条，用户发现不了
```

### Agent Bus（消息总线）

```
通俗理解：多个 Agent 之间不直接互发消息，而是通过一个共享区域读写
你的项目：agent_bus/tasks/*.md（GPT写槽位→Claude读槽位→各写各的）
为什么需要：直接对话丢上下文。共享文件是唯一事实源，每个Agent只读写自己的槽位
不用会怎样：你在GPT和Claude之间来回复制粘贴，还容易漏信息
```

### Multi-Agent Workflow（多智能体工作流）

```
通俗理解：多个专长不同的 AI 按顺序协作完成一个复杂任务
你的项目：你（目标）→ GPT（分析）→ Claude（执行）→ GPT（评审）→ 你（决策）
为什么需要：没有单个 AI 能同时做好战略判断和代码实现。分工比全能更可靠
不用会怎样：一个 AI 既做方向又做实现，判断和执行的错误互相放大
```

### Knowledge Management（知识管理）

```
通俗理解：把项目中的发现、决策、规律结构化存下来，下次不重新踩坑
你的项目：research/insights/ + research/insights/ + decisions/ + research/insights/ + research/questions/
为什么需要：一个项目的经验不沉淀，下个项目从零开始。认知迭代越快越需要冻结
不用会怎样：每个月都在重新"顿悟"同样的东西
```

### Learning Loop（学习回路）

```
通俗理解：用户反馈 → 提炼规律 → 验证 → 沉淀 → 指导下一次决策
你的项目：反馈系统（6步选择题）→ experiments/ → research/insights/ → 第二个项目验证
为什么需要：不靠猜需求，靠反馈数据。数据越多决策越准，形成飞轮
不用会怎样：不停加功能，不知道用户到底在乎什么
```

### Context Engineering（上下文工程）

```
通俗理解：不是改模型本身，而是精心设计给模型的输入来获得更好的输出
你的项目：提示词里的法条硬性要求、[第X段]编号标记、结构化摘要四段式指令
为什么需要：模型能力不变的情况下，提示词设计决定输出质量
不用会怎样：AI 输出格式混乱、不引用法条、无法溯源验证
```

### Tool Calling（工具调用）

```
通俗理解：AI 不只能生成文字，还能调用外部函数（查数据库、发请求、读文件）
你的项目：目前未使用。判例助手是先查法条库再让AI分析，而不是让AI自己决定何时查
为什么需要：让 AI 自己能查资料、调API，减少预编排的工作量
不用会怎样：编排逻辑全在人手里，灵活度受限于预设流程
```

### Agent（智能体）

```
通俗理解：能自主决策"下一步做什么"的 AI，不只是回答问题
你的项目：你+GPT+Claude 合在一起是一个 Agent 系统，但单个模型不是 Agent
为什么需要：Agent 能处理开放式任务（"帮我研究这个法律问题"），而非只能走预设流程
不用会怎样：所有任务必须事先定义好步骤，处理不了"我没见过的情况"
```

### API Gateway / MCP（模型上下文协议）

```
通俗理解：统一的接口层，让不同 AI 用同一种方式访问外部工具和数据
你的项目：目前未使用。你的"接口"是 agent_bus/ 文件系统
为什么需要：换一个模型不用重写所有工具接入逻辑
不用会怎样：每换一个模型就要重写一套调用方式
```

## 三、以后看到新名词怎么快速分类

| 听到的词 | 问自己 | 对应你的哪层 |
|----------|--------|-------------|
| LangChain, LangGraph | 它是管什么的？ | 编排层 → 对标你的 /analyze + ThreadPoolExecutor |
| CrewAI, AutoGen | 多Agent怎么协作？ | 工作流层 → 对标你的 GPT→Claude→你 |
| MCP, API Gateway | 工具怎么接入？ | 技能层接口 → 你的 skills/ 如果对外暴露API就是这一层 |
| Pinecone, Chroma | 向量存哪？ | 知识层存储 → 你的 data/laws/ 如果换成语义搜索 |
| Agent SDK | Agent脚手架？ | 编排+技能层的胶水 → 可能替代你手写的调度逻辑 |
| Prompt Engineering | 提示词怎么写？ | 技能层 → 你的 法条提示 变量 |

**核心判断规则**：先问"这东西管什么层"，再问"跟我项目里哪个东西对标"。90% 的新名词都能这样消解。
