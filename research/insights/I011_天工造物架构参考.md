## 二、核心架构：三层角色分离

这是整个系统最值钱的设计决策——**不让任何一个组件同时拥有理解和执行的权限。**

```

└──────┬──────────────┬────────────────────────┘
       │              │
       ▼              ▼
┌──────────────┐ ┌─────────────────────────┐
│   Planner    │ │  Code-X 外骨骼（执行器）    │
│   只建议     │ │  - 读库/定位/快照           │

                 └─────────────────────────┘
                          │
              ┌───────────┴───────────┐
```

## 三、Code-X：15步标准流水线（完整状态机）

每一步有明确的前置状态和输出状态。LLM 在每步之间做裁决，不是全自动。

### 标准流水线

```
Step 1:  code_x_skill_guide     → 加载当前工作流
Step 2:  project_rules_reader   → 读取项目规则
Step 3:  repo_map               → 压缩理解仓库结构
Step 4:  issue_to_file_localizer / semantic_code_search → 定位文件
Step 5:  file_to_symbol_localizer / symbol_to_line_localizer → 定位符号与行号
Step 6:  workspace_snapshot     → 写入前快照 ⚠️ 所有写入前强制
Step 7:  patch_plan_generator   → 生成 patch 计划（LLM审阅）
Step 8:  edit_unit_planner      → 拆分 edit unit
Step 9:  conflict_detector      → 检查冲突/路径/覆盖风险
Step 10: unified_diff_generator → 给 LLM 审阅 diff
Step 11: workspace_patch_applier → 写入 workspace
Step 12: python_quality_runner / pytest_runner / lint_runner → 验证
Step 13: failure_attribution_analyzer → 失败时归因（不跳过）
Step 14: repair_loop_planner → next_patch_generator → 回到 Step 7
Step 15: handoff_digest → zip_delivery_packager → 交付
```

### 任务类型分支

不同任务走不同的工具子集：

**Bug修复**：`guide → rules → repo_map → localizer → symbol → snapshot → plan → edit_unit → conflict → apply → quality → pytest → (归因→修复→循环) → digest`

**Import错误**：`repo_map → trace_mapper → import_analyzer → localizer → affected_area → snapshot → plan → apply → quality → pytest → digest`（多了 import 专项分析）

**新功能**：`rules → repo_map → semantic_search → affected_area → test_design_subagent → snapshot → plan → apply → pytest/build → review_subagent → digest`

**多文件重构**：`repo_map → dependency_graph → call_graph → affected_area → refactor_review_subagent → snapshot → plan → apply → pytest/build/typecheck → index → digest`

### 工具选链最小集（LLM决策参考表）

| 工具 | 触发条件 | 必要输入 | 下一步 |
|------|---------|---------|--------|
| project_rules_reader | 真实仓库任务开始前 | repo_root | repo_map |
| repo_map | 首次理解仓库/定位前/重构前 | workspace_root | localizer |
| issue_to_file_localizer | bug/需求/报错定位文件 | issue_text | symbol_localizer |

## 四、心流周天：失败恢复闭环（五步法）

**源码证实**：`tiangong_xinliu_zhoutian.py` — 5个工具注册入口，每个有明确的输入schema和可用性声明。

### 五步流水线

```
patch执行 → 验证失败
         ↓
   ① 观劫（guanjie）    失败归因分析 — 只读分析劫痕，不补代码
         ↓              输入：trace + stdout + stderr + returncode + changed_files
         ↓              输出：归因结论
   ② 返照（fanzhao）    patch静态审查 — 阻断伪通过、删安全测试、弱安全边界
         ↓              输入：diff + patch_summary + operations + changed_files  
         ↓              输出：passed? + decision（block/allow）
   ③ 补劫（bujie）      修复决策 — 连续失败/新增失败扩大时止乱
         ↓              输入：归因分析 + 返照结果 + repeated_failure_count + new_failures
         ↓              输出：continue_allowed? / decision（finish_and_report/rollback）
   ④ 内景录（neijinglu） 结构化交付报告 — 未完整全量pytest不声明通过
         ↓              输入：objective + 修改清单 + 验证结果 + 返照 + 补劫
         ↓              输出：结构化报告（含rollback_reference + undone + unverified）
   ⑤ 周天循环（zhoutian）主循环决策 —
                        输入：返照→观劫→补劫→内景录 全链追踪
                        输出：ok? + decision（blocked_by_fanzhao/continue/...）
```

### 关键设计原则

1. **观劫只读不修** — 归因和修复是两个独立步骤，归因阶段不能碰代码
2. **返照先过安全闸** — 任何 patch 在写入前都要检查：是否删了安全测试？是否弱化了安全边界？
3. **补劫有上限** — 连续失败不是无限修，"连续失败或新增失败扩大时止乱"——治乱不治单个 bug
4. **内景录不全量不声称通过** — 没跑完整 pytest 不能说"已验证"，和 I010 的"伪校验"是同一个病
5. **降级退出永不锁死** — 修不好就回滚、降级、mark undone，不留死锁状态

### 注册机制（5个工具的统一注册模式）

```python
_register(name, handler, description, properties)
# 每个工具：名称 + 处理器函数 + 描述 + JSON Schema + 可用性声明 + 健康检查
# 统一注册到 zhucebiao（工具注册表）
```

## 五、天搜（TianSou）：中文优先工程检索

- 中文 2-4 字 ngram

- 不执行外部搜索工具
- 不执行本地命令
- 不写源码/不删除/不提权/不外传密钥

## 七、安全和治理

### A0-A5 风险分级
- A0-A4：默认允许 + 审计记录
- A5：**硬阻断** — 密钥泄露 / 代码外传 / 大规模删除 / 系统目录破坏 / 生产环境修改 / 恶意命令

### 治理规则
- 代码任务禁止 Planner 或子代理夺权
- 子代理不得直接提交主 patch
- 验证失败必须进入归因，不能跳过
- 普通任务最多 3 轮修复循环，长链任务 6 轮
- 超限后输出阻断原因+已改文件+测试结果+下一步建议

## 九、不该学的东西

1. **修真命名** — "观劫/返照/补劫/内景录/周天循环/心流/临渊者"是工程流水线套了道教包装。用工程术语。
2. **GPT-5.5 吐的架构自己不懂** — 开发者十天从零到交付，但 GPT 排的步骤他不一定理解。你的慢路（自己拆骨架）才是你的护城河。
3. **版本号膨胀** — 0.65→2.0 十天走完，版本号不代表成熟度。真实成熟度看 test 覆盖率和真实用户数。
