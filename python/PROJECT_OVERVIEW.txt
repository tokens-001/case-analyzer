判例助手 —— 项目全貌

【定位】

不是通用AI聊天工具，是专精法律判例分析工具。核心竞争力不在"接了什么模型"，而在AI之外的硬校验层——法条有效性、溯源可靠性、判例关联网络——这些是通用AI做不到的事。

技术栈：Python Flask + DeepSeek API + 原生HTML/CSS/JS，部署在 PythonAnywhere。

【已实现功能（按层次）】

第一层：AI分析（基础能力）
- 四维并行分析：核心争议、推理链路、未回答问题、可平移性，线程池同时调用API
- 综合总结：四段分析整合为300字以内一段总结
- 法条主动引用：提示词硬性要求AI写明《XX法》第X条，至少2条，不引用视为不合格
- 文件上传：支持粘贴文字或上传txt文件

第二层：溯源与校验（硬能力层，通用AI做不到的）
- 智能分段：空行→换行→句号三级自动分段，发给AI前插入[第X段]编号标记，保证两边对齐
- 溯源对比：AI必须标注(见原文第X段)，代码用同一套分段函数校验，越界标红，防止AI编造引用
- 法条统计：通用正则匹配任意《XX法》第X条，不限于特定法律
- 法条库查询：从data/laws/读取法条原文对照AI引用
- 法条有效性校验：元数据标注颁布/施行/修订日期+现行有效/已修订/已废止状态，前端绿黄红三色标签
- 缺失法条自动记录：AI引用过但法条库没有的，自动写入missing_laws.txt，自增长机制
- 分析质量验证：字数过少、未引用法条等自动告警

第三层：知识网络（壁垒层）
- 判例关联图谱：扫描全部历史判例→构建法条→判例反向索引→查一个判例时自动找出引用相同法条的其他判例，按关联强度排序
- 关联判例跳转：点击关联判例直接加载完整分析

第四层：用户体验
- 会话隔离：每个浏览器UUID，数据互不干扰
- IP+UID双轨限流：每人每日默认3次，IP+Cookie双重记录防绕过
- 剩余次数显示：页面加载时查询，每次分析后更新，用完变红
- 历史记录：自动保存，只显示案件名+日期，点击展开完整四维分析
- 搜索历史：关键词实时过滤
- 下载报告：一键导出txt，含分析+溯源+法条对照
- 暗色主题：专业深色UI，手机端响应式单列

第五层：基础设施
- dotenv自动加载.env文件
- GitHub同步：本地改→push→PythonAnywhere pull→重启
- 法条格式规范：统一元数据+条文格式

【当前数据资产】

- 法条库：1部法律（民法典侵权责任编，5条条文），格式规范已建立
- 判例库：用户分析过的判例自动积累JSON文件
- 缺失法条：每次分析自动记录，指引后续扩充方向

【发展方向（三阶段）】

1. 法条库做实（已完成框架，内容持续扩充）：元数据规范+有效性校验+自增长机制
2. 判例关联图谱（刚完成）：法条同现反向索引，自动发现关联判例
3. 法院裁判画像（待做）：同一法条在不同法院的适用差异统计、裁判倾向分析

【核心理念】

AI加功能是横向加法，谁都能做。专精是纵向减法——砍掉泛的，把AI之外的硬校验做到别人没法复制。

三个壁垒：
- 法条时效性：AI不知道哪条法废止了，我知道
- 溯源可验证：AI会编造引用，我逐条对原文
- 判例关联网：AI只看单案，我能串出法条→判例的关系网

【当前局限与待解决】

1. 法条库内容太少（1部法律），需持续扩充
2. 没有外部判例数据库接入（北大法宝API预留但未接）
3. 判例量不够做统计意义的法院画像
4. 无用户反馈机制（分析质量评分等）
5. 搜索只有关键词匹配，无法语义搜索
6. 关联图谱只基于法条同现，未考虑争议类型相似度
