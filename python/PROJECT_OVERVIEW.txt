判例助手 v3 —— 项目全貌

【定位】

不是通用AI聊天工具，是专精法律判例分析工具。核心竞争力不在"接了什么模型"，而在AI之外的硬校验层——法条有效性、溯源可靠性、判例关联网络、反例发现——这些是通用AI做不到的事。

产品理念已从"帮律师找案例"升级为"法律AI审计员"——不是比谁分析得多，而是比谁能证明自己可信、能主动指出哪里不可信。

技术栈：Python Flask + DeepSeek API + 原生HTML/CSS/JS，部署在 PythonAnywhere。

【已实现功能（按层次）】

第一层：输入校验（入口层）
- 判例文字粘贴/上传：textarea粘贴 + 上传txt自动填充文件名
- 判例名称必填：前后端双重校验，空名称拒绝，最多80字
- 案发日期必填：HTML date选择器 + required属性，后端验证
- 判例文字最少50字校验
- 清空按钮一键重置

第二层：AI分析（九维并行，分两轮ThreadPoolExecutor调用）

第一轮4路并行：
- 核心争议：案件的核心法律争议焦点分析
- 推理链路：法院从事实到结论的推理过程
- 未回答问题：判决中未解决的法律问题
- 可平移性：分析框架可推广到哪些法律领域

第二轮5路并行（依赖第一轮结果）：
- 综合总结：四段整合为300字以内一段总结
- 反例检索：主动寻找反对观点、相反判例倾向、三个最强反驳论点
- 判决结构化摘要：事实认定→争议焦点→裁判理由→裁判结果四段式拆解
- 论证链检测：事实→法条→结论映射检测，找逻辑断裂和未证明前提，提供替代论证路径
- 证据缺失检测：指出缺少哪些关键证据，补充哪些最可能改变判决结果
- 相反法条发现：找出可能限制/抵消/冲突本案适用法条的其他条文

提示词硬性要求AI写明《XX法》第X条，至少2条。发送前统一插入[第X段]编号标记保证对齐。

第三层：硬校验层（通用AI做不到的）

溯源校验：
- 智能分段：空行→换行→句号三级自动拆分
- 溯源对比：AI必须标注(见原文第X段)，代码用同一套分段函数校验，越界标红，防止AI编造引用
- 编造检测：段落号超出原文范围自动判定

法条校验：
- 法条统计：通用正则匹配任意《XX法》第X条，不限于特定法律名称
- 法条库查询：从data/laws/读取法条原文对照AI引用
- 法条有效性标签：元数据标注颁布/施行/修订日期+现行有效(绿)/已修订(黄)/已废止(红)
- 法条版本时间轴：案发日期(必填) vs 法条施行/修订日期比对，版本不匹配标红警告
- 相邻法条发现：AI引用法条的前后±2条自动扫描展示（前条蓝/后条紫标签）
- 缺失法条自增长：AI引用过但法条库没有的，自动写入missing_laws.txt

分析质量验证：
- 字数检查：每个维度字数过少判定API异常
- 法条引用检查：四段均未引用法条→告警
- 可信度评分：综合法条版本+溯源+验证，0-100分+高/中/低三级
- 可信度仪表盘：圆环分数+等级+加减分明细来源
- 风险列表：三级汇总（验证+法条版本+溯源越界），红色danger/黄色warning，前端优先展示

第四层：知识网络（壁垒层）
- 判例关联图谱：扫描全部历史判例→构建法条→判例反向索引
- 关联判例发现：查一个判例自动找引用相同法条的其他判例，按共同法条数量排序
- 关联判例跳转：点击直接加载完整九维分析
- 历史记录：自动保存JSON，列表显示案件名+日期+总结摘要
- 历史搜索：关键词实时过滤判例名和总结
- 历史详情回看：点击加载完整九维分析（含关联判例）

第五层：输出与交互（体验层）
- 风险优先展示：打开结果先看到"哪里可能错"，不是先看分析卡片
- 可信度仪表盘：圆环分数+等级文字+加减分明细
- 判决结构化卡片：四段式摘要，紫色左边框独立卡片
- 2×4九维卡片：每张顶部3px色条区分维度（蓝紫琥珀青绿红橙粉），hover边框高亮+阴影
- 综合总结卡片：绿色左边框独立展示
- 相邻法条区：前条蓝色标签/后条紫色标签，引用法条名+条文摘要
- 折叠区：溯源对比/法条库对照/关联判例默认折叠，点击箭头展开
- 结果渐入动画：卡片依次入场，级联延迟
- 下载报告：一键导出13段完整txt（九维分析+法条统计+法条对照+溯源对比）
- 响应式布局：手机端(<700px)自动单列
- 剩余次数显示：页面加载查询+每次分析后更新，用完变红

第六层：基础设施
- 会话隔离：每个浏览器UUID，数据互不干扰
- IP+UID双轨限流：每人每日默认3次（DAILY_LIMIT环境变量可调），双维度防绕过
- X-Forwarded-For兼容PythonAnywhere代理
- dotenv自动加载.env文件
- GitHub同步：本地改→push→PythonAnywhere pull→重启
- JSON存储：每次分析存为 {uid}/{日期}_{判例名}.json
- 法条格式规范 LAW_FORMAT.md：统一元数据+条文格式
- 代码地图 CODE_MAP.md：前后端树状架构速查
- 名词速查 GLOSSARY.md：15类145条编程+AI+法律专业名词
- 功能树 功能树.md：7层37项功能的完整清单

【网页视觉设计】

整体风格：专业法律暗色主题，GitHub风格深色界面。

配色体系：CSS变量驱动6色语义系统——蓝色(核心争议)、紫色(推理链路)、琥珀色(未回答问题)、青绿色(可平移性)、绿色(总结/验证通过)、红色(反例检索/风险警告)、橙色(论证链检测)、粉色(证据缺失)、青色(相反法条)。底色#0d1117，表面层#161b22，边框#30363d。

卡片设计：2×4网格布局，每张顶部3px色条标识维度。反例/论证链/证据缺失/相反法条卡片红色/橙色/粉色/青色边框区分。hover边框提亮+阴影。结构化摘要紫色左边框，综合总结绿色左边框。

头部：Logo区渐变色方块(蓝→紫)+天平图标，标题加粗，一行标语。

动效：结果区逐项渐入动画(fadeInUp，子元素级联延迟)。按钮hover微上移+投影，点击回弹。加载spinner。风险项单独入场动画。

输入区：判例文字等宽字体，焦点蓝色光环。按钮渐变(蓝→靛)+hover投影+active回弹。案发日期input与文本输入风格统一。

响应式：700px断点以下卡片变单列，间距自适应。

【核心理念】

AI加功能是横向加法，谁都能做。专精是纵向减法——砍掉泛的，把AI之外的硬校验做到别人没法复制。

四个壁垒：
- 法条时效性：AI不知道哪条法废止了、版本对不对，我知道
- 溯源可验证：AI会编造引用，我逐条对原文并标红越界
- 反例发现：大多数AI工具只找支持你的案例，我找反对你的案例
- 判例关联网：AI只看单案，我能串出法条→判例→裁判结果的关系网
