判例助手 v2 —— 项目全貌

【定位】

不是通用AI聊天工具，是专精法律判例分析工具。核心竞争力不在"接了什么模型"，而在AI之外的硬校验层——法条有效性、溯源可靠性、判例关联网络、反例发现——这些是通用AI做不到的事。

产品理念已从"帮律师找案例"升级为"帮律师发现自己论证链中的漏洞"。

技术栈：Python Flask + DeepSeek API + 原生HTML/CSS/JS，部署在 PythonAnywhere。

【已实现功能（按层次）】

第一层：AI分析（基础能力）
- 六维并行分析：核心争议、推理链路、未回答问题、可平移性、综合总结、反例检索，线程池同时调用API
- 反例检索：AI主动寻找反对观点、相反判例倾向、论证薄弱环节，对方律师最有力的三个反驳论点
- 判决结构化摘要：事实认定→争议焦点→裁判理由→裁判结果四段式拆解
- 法条主动引用：提示词硬性要求AI写明《XX法》第X条，至少2条，不引用视为不合格
- 文件上传：支持粘贴文字或上传txt文件
- 案发日期输入：可选填，用于后续法条版本校验

第二层：溯源与校验（硬能力层，通用AI做不到的）
- 智能分段：空行→换行→句号三级自动分段，发给AI前插入[第X段]编号标记，保证AI和代码用同一套分段标准
- 溯源对比：AI必须标注(见原文第X段)，代码用同一套分段函数校验，越界标红，防止AI编造引用
- 法条统计：通用正则匹配任意《XX法》第X条，不限于特定法律名称
- 法条库查询：从data/laws/读取法条原文对照AI引用
- 法条有效性校验：元数据标注颁布/施行/修订日期+现行有效/已修订/已废止状态，前端绿黄红三色标签
- 法条版本时间轴：输入案发日期→自动比对法条施行/修订日期→版本不匹配标红警告（如案发于2021年但AI引用2025修订版条文）
- 缺失法条自动记录：AI引用过但法条库没有的，自动写入missing_laws.txt，自增长机制
- 风险列表生成：验证+法条版本+溯源越界三级汇总，前端优先展示
- 分析质量验证：字数过少、未引用法条等自动告警

第三层：知识网络（壁垒层）
- 判例关联图谱：扫描全部历史判例→构建法条→判例反向索引→查一个判例时自动找出引用相同法条的其他判例，按共同法条数量排序
- 关联判例跳转：点击关联判例直接加载完整分析，附带共同法条标签

第四层：用户体验
- 风险优先展示：打开结果先看到"哪里可能错"（红色危险/黄色警告），不是先看分析卡片
- 会话隔离：每个浏览器UUID，数据互不干扰
- IP+UID双轨限流：每人每日默认3次，IP+Cookie双重记录防绕过
- 剩余次数显示：页面加载时查询，每次分析后更新，用完变红
- 历史记录：自动保存，只显示案件名+日期，点击展开完整六维分析
- 搜索历史：关键词实时过滤判例名和总结
- 下载报告：一键导出txt，含分析+反例+结构化摘要+溯源+法条对照
- 响应式布局：手机端自动单列

第五层：基础设施
- dotenv自动加载.env文件
- GitHub同步：本地改→push→PythonAnywhere pull→重启
- 法条格式规范 LAW_FORMAT.md：统一元数据+条文格式
- 代码地图 CODE_MAP.md：前后端树状架构速查，每个模块标行号
- 名词速查 GLOSSARY.md：13类118条编程+AI专业名词，每条标注代码位置

【网页视觉设计】

整体风格：专业法律暗色主题，GitHub风格深色界面。

配色体系：CSS变量驱动的6色语义系统——蓝色（核心争议）、紫色（推理链路）、琥珀色（未回答问题）、青绿色（可平移性）、绿色（总结）、红色（反例检索/风险警告）。底色 #0d1117，表面层 #161b22，边框 #30363d。

卡片设计：六张分析卡片2×3网格布局，每张顶部3px色条标识维度类别。反例卡片红色边框区分，hint其"反对观点"语义。hover时边框提亮+微阴影。结构化摘要紫色左边框突出。

头部：Logo区渐变色方块（蓝→紫）+天平图标⚖️，标题加粗，一行标语说明定位。

动效：结果区逐项渐入动画（fadeInUp，子元素级联延迟）。按钮hover微上移+投影，点击回弹。加载spinner精致化。风险项单独入场动画。

输入区：判例文字用等宽字体，焦点时蓝色光环。按钮渐变（蓝→靛）+hover投影+active回弹。案发日期input与文本输入风格统一。

响应式：700px断点以下卡片变单列，间距自适应。

【已实现但需持续积累】

- 法条库：1部法律（民法典侵权责任编，5条条文），格式规范已建立，内容靠手动核验+missing_laws.txt自增长驱动
- 判例库：随使用自动积累JSON文件
- 判例关联图谱：依赖判例量，量越大关联越丰富

【未实现 / 待做】

短期可做（不需要大量数据）：
1. 法条库内容扩充——目前只有1部法律，需从flk.npc.gov.cn等官方来源手动核验补入高频条文
2. 语义搜索——目前历史搜索只支持关键词匹配，可用embedding做语义相似搜索
3. 用户反馈机制——分析质量评分、纠错入口

中期（需要积累判例量）：
4. 判例衰减机制——案例影响力/权威性/时效评分，标注已被新司法解释覆盖或指导案例推翻的判例
5. 争议焦点树——将案例按争议类型归类为树状结构（如"劳动关系认定→是否存在人身依附性"），挂载相关判例
6. 裁判规则抽取——从多个判例中提取隐含规则（如"当A+B+C条件满足时，支持劳动关系的置信度87%"）

长期（需要大量数据+外部API）：
7. 外部判例数据库接入——北大法宝API预留但未接
8. 法院裁判画像——同一法条在不同法院的适用差异、单个法官的裁判倾向（劳动者保护倾向、证据要求严格度、平均判决周期等）
9. 法条实效性统计——某法条引用案件数、支持率、平均赔偿区间

【核心理念】

AI加功能是横向加法，谁都能做。专精是纵向减法——砍掉泛的，把AI之外的硬校验做到别人没法复制。

四个壁垒：
- 法条时效性：AI不知道哪条法废止了、版本对不对，我知道
- 溯源可验证：AI会编造引用，我逐条对原文并标红越界
- 反例发现：大多数AI工具只找支持你的案例，我找反对你的案例
- 判例关联网：AI只看单案，我能串出法条→判例→裁判结果的关系网

【上轮GPT反馈后的落地情况】

GPT提的四个高价值方向已全部实现：
✅ 法条版本时间轴（案发日期 vs 法条修订日期比对）
✅ 反例检索（第六维分析，主动找反对观点）
✅ 判决结构化（事实→争议→理由→结果四段式）
✅ 风险优先展示（先报错再展示分析）

另有GPT提的方向因数据量不足暂未做：判例衰减机制、争议焦点树、裁判规则抽取、法官认知画像。
