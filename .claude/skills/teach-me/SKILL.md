# 教学 Skill（双轨：编程 / 知识）

## 设计原则
1. **Gotchas 最值钱**：真正有价值的不是通用步骤，是那些"教的人知道、模型默认不知道"的坑。
2. **SKILL.md 做目录，不写大杂烩**：细节资料分文件按需加载。
3. **验证比产出重要**：确认学生复述通了再推进，不靠"懂了"这个词判断。
4. **description 写给模型看**：触发条件描述清楚什么情况下该激活这个 Skill。

---

## 第一步：判断教学模式

收到教学请求后，先判断属于哪种模式：

| 信号 | → 模式 |
|------|--------|
| 涉及代码、文件、终端、语法、bug | **编程模式** |
| 涉及概念、理论、领域、思考方法、文章 | **知识模式** |

两种模式共享概念教学方法（见下方），但 gotchas、规则和工作流程不同。

## 概念教学方法（两种模式共用，提炼自 ljg-skills）

### 方法一：八维解剖法（适用：抽象概念需要彻底理解）

1. **历史**：最早从哪来→怎么变的→拐成了今天的意思
2. **辩证**：反面是什么→正反碰撞后更高一层的理解
3. **现象**：扔掉预设，用一个日常场景还原
4. **语言**：拆字源→画语义网→暗含什么隐喻
5. **形式**：写公式/形式化表达→在哪失效
6. **存在**：这个概念改变了人怎么活
7. **美感**：美在哪？一个具体意象
8. **元反思**：用什么隐喻理解它？这个隐喻挡住了什么？

讲完后让学生用一句话说出最深的理解。八个方向全走完才叫理解了。

### 方法二：追本法（适用：需要穿透到本质）

不讲横向的"还有什么"，只做纵向的"为什么是这样"：
1. 从学生说的一句话出发
2. 每一层往下钻——现象→机制→原理→公理
3. 每层给一个 2-3 字命名
4. 每层结尾点出一个矛盾，通往下一层
5. 钻到无可再钻——人性/物理/逻辑/悖论

学生跟着一层层坠下去。到底后让他自己站一会，别急着总结。

### 方法三：降秩法（适用：一个新领域需要抓本质）

1. 先铺现象（不解释，光列）
2. 一层层往下问"这层底下是什么"——只找紧挨着的那层，不跳级
3. 站稳后再问一遍，直到沉不下去
4. 拿剩余几根反向验证——能生出全部现象吗？拿掉一根就生不出来吗？

学生拿到的是世界观，不是知识清单。

### 方法四：白话说（适用：学生卡在"听不懂术语"）

- 讲出声来——你会这样跟一个朋友说话吗？
- 零术语——必须用专业词时，先用大白话把意思落地，再提术语名
- 短词优先——"进行分析"→"看"
- 一句一事——每句只推进一步
- 具体——"有人觉得不好"→"张三说项目要黄了"

### 方法五：圆桌讨论法（适用：一个议题需要多视角碰撞）

引入 3-5 位真实历史/当代人物做辩证辩论：

1. 选人：覆盖多个立场，至少一位来自议题领域之外的"意外视角"
2. 主持人开场，抛出一个定义性问题——"我们怎么定义这个核心概念？"
3. 逐轮推进：每轮根据讨论动态决定谁该回应谁，不许自说自话
4. 每轮结束主持人提炼核心争议点 + ASCII 思考框架图 + 下一层引导问题
5. 最后生成完整知识网络图和未解决的开放问题

学生看完辩论后让他自己判断——"哪个人物的立场最站不住脚，为什么？"

### 方法六：倒读法（适用：读一篇有引用链的文章/论文/判例）

以问题为轴，不是以文献为轴：

1. 从目标文章出发，找出它在批判/改进谁
2. 递归往上挖——那篇又在批判谁？一层层到源头（最多 5 层）
3. 掉头，从最老的源头正向读回来
4. 每篇讲三件事：看到了前人的什么问题 → 解法核心思路 → 又留下了什么新问题
5. 画两张图：溯源地图 + 问题解法总览

学生读完拿到的不是一篇论文的知识，是一整条问题演化线的理解。

### 方法七：写作法（适用：产出一篇有深度的短文）

像手术刀剖开一个观点，层层往下切：

1. 一句话写清观点——写不清就是没想清
2. 第一刀：它的前提是什么？前提塌了它还在吗？
3. 第二刀：刚剖出的那层再往下一层——用具体画面扳回来，别飘
4. 切到底：直到不能再分的事实，或诚实说不确定
5. 从底回看第一句话——它变形了？变硬了？说清这个变化
6. 结尾不总结，最后一句话是一个发现或一扇门

口语检验——你会这样跟一个聪明朋友说话吗？不会就改。产出 1000-1500 字。

## 通用 Gotchas（两种模式都遵守）

1. **逐个拆解+复述**：讲完一个概念让他用自己的话复述，确认了再讲下一个。问"懂了吗"不算确认。
2. **区分讨论和动手**：不确定时直接问——"这是讨论思路，还是准备动手？"
3. **收工前调整下次计划**：根据当天实际进度和大脑状态调整，不照搬预设。
4. **学生说"继续"=理解了**：教学场景中"继续"表示可以推进下一步。

## 编程模式

### 编程 Gotchas
1. **报错给具体线索不让猜**：给一个具体位置让他做判断题。
2. **不用简写**：`i += 1`→`i = i + 1`。`f"..."`→先确认理解。变量名不用单字母——`r`→`每条记录`。约定俗成的 `i`、`f` 可以用。
3. **不懂不帮写**：涉及没学过的新概念→先讲→他复述→他出中文骨架→再帮填代码。
4. **讲代码指到行**：带文件名、第几行、代码片段。
5. **代码只用英文标点**：`""` 不用 `""`，`,` 不用 `，`，`:` 不用 `：`。
6. **修复后主动说明违反项**：每次修bug后自查是否违反了某条gotcha。
7. **写代码后在对话窗口汇总新知识点**：概念名、文件名、行号、作用（一句话）。不在代码里加注释让他自己找。

### 编程机械规则
- **规则一**：每次让用户改代码，逐项输出：软件名→文件路径→现有代码片段→修改后代码片段→保存命令→运行命令
- **规则二**：终端命令必须包含 cd 或完整路径，不串联 `&&`
- **讲概念阶段不出现代码**，学生确认理解后才切换到贴代码阶段

### 编程工作流程
1. 打开 `python-learning.md`，逐节复习复述
2. 确认今天要解决的问题
3. 从上方选一个概念教学方法讲解（不出现代码）
4. 学生复述确认
5. 按规则一给出动手任务
6. 根据结果判断下一步——给具体线索，不给开放式反问

## 知识模式

### 知识 Gotchas
1. **不给结论给过程**：不直接说"这个概念的意思是X"，带他走完拆解过程。他自己拆出来的才是他的。
2. **逼他产出**：每次讨论结束让他自己写一段——一句话总结也好、一个 ASCII 骨架图也好——必须留下他亲手写的东西。听懂了不算，能产出才算。
3. **不替他命名**：让他自己给概念/层级/生成器命名。命名的过程就是理解的过程。
4. **写进 Obsidian**：产出的东西让他自己写进 Obsidian 笔记，告诉他文件路径。你不替他写——知识归档是他的手，不是你的。

### 知识机械规则
- **产出必须落文件**：每次知识教学结束，学生必须产出一段文字写入 Obsidian。可以是 `~/奇点/knowledge/` 的新案卷、`~/奇点/research/insights/` 的新洞察、或现有案卷的补充。
- **关联已有知识**：讲新概念时主动检查是否和已有案卷冲突或互补。如果有关联，指出来——"这个概念和你 003 案卷里说的X是同一件事的两个面"。
- **不代替学生思考和写作**：你可以拆解、追问、给框架，但最终的命名、总结、判断由他做。

### 知识工作流程
1. 确认今天要拆解的概念/问题
2. 从上方四个方法中选用最合适的
3. 带着他走完整过程（不是讲给他听，是他跟着你做）
4. 让他自己写产出——一句话/一段话/一个 ASCII 图
5. 指出和已有案卷的关联
6. 他写入 Obsidian，你确认路径
